# Lead Processor — опис рішення

## Що це

MVP-бекенд, що обробляє заявку з форми лендингу: `POST /api/leads` приймає JSON і проганяє його через
пайплайн `валідація → нормалізація → AI-summary + класифікація → запис у SQLite → сповіщення в Telegram`,
повертаючи структурований результат. Це прототип, щоб показати підхід і якість, а не production-ready система.

---

## Підхід і ключові рішення

- **MVP без overengineering:** без черг, воркерів, мікросервісів; синхронна відповідь.
- **Чіткий поділ помилок:** критичні (невалідний вхід → `422`; збій запису в БД → `5xx`) зупиняють обробку;
  некритичні (AI, Telegram) ловляться всередині сервісів і дають фолбек — пайплайн ніколи не «падає» через них.
- **Семантика AI-результату:** `ai_ran=false` (AI вимкнено або інфра-збій) відрізняється від
  `category="unknown"` (AI відповів, але вміст незрозумілий) — обидва стани зберігаються до БД і у відповіді.
- **Конфіг через `.env` (pydantic-settings), fail-fast:** ключ/токени потрібні лише коли відповідна фіча
  увімкнена; прапорці `AI_ENABLED`/`TELEGRAM_ENABLED=false` дають запуск без зовнішніх сервісів.
- **Без вигаданих API:** Gemini через офіційний `google-genai` (модель `gemini-2.5-flash`), Telegram через
  Bot API (`httpx`), сховище — stdlib `sqlite3` (файл у проєкті, без зовнішньої інфраструктури).
- **Якість:** type hints на межах модулів, `ruff` + `mypy` чисті, `pytest` (50 тестів).
- **Модульність за відповідальністю:** `config` / `schemas` / `services` / `pipeline` / `routes` / `logger`.
